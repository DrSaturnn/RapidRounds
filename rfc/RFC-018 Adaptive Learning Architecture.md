# RFC-018: Adaptive Learning Architecture

## Status

Architectural RFC. This document defines how RapidRounds transforms learner
evidence into educational actions. It does not introduce runtime behavior,
database schema, API contracts, scoring formulas, or implementation-specific
code.

RFC-017 defines what RapidRounds knows about a learner. RFC-018 defines how
RapidRounds uses that knowledge.

This RFC consolidates the architectural responsibilities previously envisioned
for an Adaptive Decision Engine, Decision Object Architecture, Curriculum Graph
interaction, and retrieval/review strategy. It is the umbrella adaptive
architecture for RapidRounds.

## Context

RapidRounds is an adaptive clinical reasoning engine. Its core responsibility
is not delivering questions; it is selecting the next highest-value educational
action based on learner evidence, learner state, and curriculum relationships.

This RFC MUST remain consistent with
[docs/PROJECT_CONSTITUTION.md](../docs/PROJECT_CONSTITUTION.md) and
[RFC-017: Canonical Learner State Engine](./RFC-017%20Canonical%20Learner%20State%20Engine.md).

## 1. Architectural Role

RFC-018 defines the adaptive layer between:

```text
Learner State
  -> Educational Evidence
  -> Adaptive Decisions
  -> Learner Experience
```

The adaptive layer interprets what RapidRounds knows about the learner and
chooses what should happen next.

Adaptation is the core responsibility of RapidRounds because the product is not
a static question delivery system. A question is one possible intervention, but
so are repair, boundary teaching, deeper explanation, prerequisite review,
delayed retrieval, and future coaching.

All adaptive behavior MUST derive from canonical learner state and curriculum
relationships. Future systems MUST NOT create feature-specific adaptive engines
that bypass this architecture.

## 2. The Adaptive Learning Loop

The canonical adaptive loop is:

1. Learner performs an educational interaction.
2. The interaction produces educational evidence.
3. Evidence updates Learner State.
4. Learner State is interpreted through the Curriculum Graph.
5. The Adaptive Learning Engine determines the highest-value educational action.
6. The learner receives that intervention.
7. The cycle repeats.

This loop is the canonical adaptive architecture for RapidRounds. It applies to
practice questions, Decision Repair, Teach Me More, Continue Learning, delayed
review, future simulations, and future AI coaching.

The loop MUST remain explainable. RapidRounds should be able to trace an
adaptive action back to learner evidence and curriculum relationships.

## 3. Educational Actions

RapidRounds adapts by selecting educational actions, not merely questions.

Educational actions may include:

- new RapidRound
- Decision Repair
- Decision Boundary Repair
- Teach Me More
- Continue Learning
- delayed retrieval
- prerequisite review
- broaden concept
- increase complexity
- decrease complexity
- future simulation
- AI coaching

Educational actions are the canonical output of adaptation because different
learner states require different interventions. An unknown response, a high
confidence misconception, a partial answer, and a repeated successful retrieval
should not all produce the same next step.

The Adaptive Learning Engine SHOULD select the smallest useful action that
advances learning while preserving momentum.

## 4. Educational Evidence

Educational evidence is an observation generated by a learner interaction.
Evidence is not the same as a conclusion.

Educational evidence may include:

- correctness
- confidence
- partial credit
- distractor selection
- reasoning quality
- clue recognition
- retrieval success
- repair completion
- latency
- repeated performance

Evidence SHOULD remain auditable and tied to the interaction that produced it.
Evidence MAY support mastery estimates, misconception tracking, repair history,
and adaptive priority, but it MUST NOT present itself as objective truth about
the learner.

This RFC does not define scoring algorithms.

## 5. Decision Objects

Decision Objects represent educational events.

Architecturally, a Decision Object standardizes the evidence produced by a
RapidRounds interaction. A Decision Object may represent a question attempt,
repair event, Teach Me More expansion, delayed review, future simulation step,
or AI coaching interaction.

Decision Objects:

- produce evidence
- standardize educational event structure
- make adaptive inputs auditable
- connect an interaction to learner state and curriculum context

Decision Objects do not own learner state. They do not own curriculum
relationships. They do not own adaptation. They do not choose the next action.

This RFC does not implement Decision Objects.

## 6. Curriculum Graph Interaction

The Curriculum Graph is the educational map. It organizes:

- concepts
- objectives
- diagnoses
- illness scripts
- clinical decisions
- prerequisite relationships
- neighboring concepts
- decision boundaries

Adaptive behavior SHOULD navigate this graph rather than isolated questions.
The graph allows RapidRounds to identify what concept was tested, what nearby
concept was confused, what prerequisite may be weak, what downstream objective
should come next, and what shelf views include the same clinical context.

The Curriculum Graph owns conceptual relationships. It MUST NOT own learner
mastery. Learner mastery belongs to Learner State.

## 7. Adaptive Decision Engine

The Adaptive Decision Engine selects the highest-value educational
intervention.

It consumes:

- Learner State
- Curriculum Graph
- Educational Evidence

It produces:

- Educational Actions

The Adaptive Decision Engine SHOULD consider learner mastery estimates,
confidence, retrieval history, repair history, cognitive error patterns,
misconceptions, recent misses, curriculum prerequisites, successor concepts,
related concepts, common distractors, and review timing.

The Adaptive Decision Engine MUST remain explainable. If it recommends a
question, repair, comparison, review, or coaching action, RapidRounds SHOULD be
able to state why that action was selected.

The Adaptive Decision Engine MUST NOT persist a competing learner model.

## 8. Retrieval and Review

Retrieval and review should emerge from learner state rather than fixed
schedules alone.

Repeated successful retrieval strengthens educational evidence. Delayed
retrieval demonstrates more durable learning than immediate repetition.
Immediate repair teaches the current boundary, while long-term review tests
whether the learner can retrieve and apply the concept later.

Review SHOULD account for concept mastery, retrieval spacing, prior repair,
confidence, related misconceptions, and curriculum position.

This RFC does not specify review algorithms, spacing intervals, or scoring
formulas.

## 9. Explainability

Explainability is a first-class architectural requirement.

RapidRounds SHOULD be able to explain:

- Why this question?
- Why this repair?
- Why this comparison?
- Why this review?
- Why now?

Explanations SHOULD be derived from learner evidence, learner state, and
curriculum relationships. Adaptive decisions MUST NOT be opaque when they affect
the learner's educational trajectory.

Learner-facing explanations should remain concise. Internal explainability may
be more detailed for auditing, analytics, and future educator tools.

## 10. Separation of Responsibilities

### Learner State

Learner State owns educational state: mastery estimates, confidence,
misconceptions, reasoning profile, retrieval history, repair history, and
longitudinal learning trajectory.

### Educational Evidence

Educational Evidence owns observations. It records what happened during an
interaction without claiming final mastery or certainty.

### Decision Objects

Decision Objects own standardized evidence events. They normalize the structure
of educational interactions so downstream systems can interpret them
consistently.

### Curriculum Graph

Curriculum Graph owns conceptual relationships: concepts, objectives,
diagnoses, illness scripts, clinical decisions, prerequisites, successors,
neighboring concepts, and decision boundaries.

### Adaptive Engine

Adaptive Engine owns educational decisions. It selects the highest-value next
educational action from learner state, evidence, and curriculum relationships.

### Question Engine

Question Engine owns presentation of questions. It should deliver the selected
RapidRound but MUST NOT decide durable learner mastery or create a separate
adaptive model.

### AI Tutor

AI Tutor owns explanations, coaching, and educational dialogue. It may generate
evidence and consume adaptive decisions, but it MUST NOT invent learner state or
override the canonical adaptive architecture.

### UI

UI owns rendering. It displays questions, repairs, recommendations, trajectories,
and explanations. It MUST NOT create adaptive decisions or durable educational
state.

No component may duplicate another component's responsibility.

## 11. Non-Negotiable Principles

1. Adaptation MUST choose educational actions, not merely questions.
2. Learner State MUST remain canonical.
3. Evidence MUST precede adaptation.
4. Educational evidence MUST be observational rather than declarative.
5. Curriculum Graph MUST remain the canonical conceptual map.
6. Adaptive decisions MUST be explainable.
7. RapidRounds MUST NOT create duplicated learner models.
8. RapidRounds MUST NOT create feature-specific adaptive engines.
9. Decision Objects MUST NOT own learner state, curriculum, or adaptation.
10. Curriculum Graph MUST NOT own learner mastery.
11. UI MUST render adaptive decisions rather than create them.
12. AI Tutor systems MUST consume the adaptive architecture rather than redefine
    it.
13. Adaptive actions SHOULD preserve practice momentum.
14. Review priority SHOULD emerge from learner state and evidence, not only
    fixed schedules.

## 12. Preparing for UI

This RFC completes the foundational adaptive architecture before UI
architecture work begins.

Future UI work should render:

- learner trajectory
- recommended action
- reason for recommendation
- mastery profile
- repair history
- concept relationships
- current learning pathway

The UI should visualize adaptive decisions rather than create them. A UI surface
may allow learner choice, but the adaptive rationale and available actions
SHOULD come from the Adaptive Learning Engine.

Future UI work SHOULD preserve the Constitution's practice-first product model:
adaptive insight should support training rather than become a dashboard-first
interruption.

## 13. Future Extensibility

This architecture naturally supports:

- adaptive scheduling
- spaced repetition
- simulations
- oral boards
- educator dashboards
- AI coaching
- Step 2 readiness prediction
- mobile offline mode

These systems SHOULD consume Learner State, Educational Evidence, Curriculum
Graph relationships, and Adaptive Decisions. They MUST NOT redefine learner
state, curriculum ownership, or adaptive decision ownership.

Future extensions SHOULD add new educational actions, evidence types, read
models, and explanation surfaces without breaking the canonical loop.

## Implementation Boundary

This RFC intentionally avoids implementation-specific code, database schemas,
API design, and scoring formulas.

Future implementation RFCs MUST preserve:

- the Learner State Engine as the canonical educational state
- Educational Evidence as observations
- Decision Objects as standardized event producers
- Curriculum Graph as the conceptual map
- Adaptive Engine as the educational decision layer
- UI as a rendering layer
- explainability as a first-class requirement
- practice flow momentum
